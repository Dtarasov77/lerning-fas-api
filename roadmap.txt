=====================================
   ROADMAP: FastAPI + Backend (ООП)
   Пошаговый план обучения
=====================================
Цель: научиться писать чистый, структурированный бекенд на FastAPI
с глубоким пониманием ООП-концепций и архитектурных паттернов.

Статусы: [ ] — не начато, [~] — в процессе, [x] — завершено


##### ЭТАП 1 — ОСНОВЫ PYTHON ООП (фундамент) #####
Без ООП невозможно писать поддерживаемый бекенд.
Здесь закладывается мышление, которое потом применяется везде.

[ ] 1.1  Классы и объекты
        - Что такое класс, экземпляр, атрибут, метод
        - __init__, self, создание объектов
        - Разница между атрибутом класса и атрибутом экземпляра
        Практика: создать класс User(name, email) с методом greet()

[ ] 1.2  Инкапсуляция
        - Публичные, защищённые (_) и приватные (__) атрибуты
        - property, getter, setter — зачем контролировать доступ
        Практика: класс BankAccount с приватным балансом и методами deposit/withdraw

[ ] 1.3  Наследование
        - Базовый класс → дочерний класс
        - super(), переопределение методов
        - Множественное наследование и MRO (порядок разрешения методов)
        Практика: BaseUser → AdminUser, RegularUser с разными правами

[ ] 1.4  Полиморфизм
        - Один интерфейс — разное поведение
        - Duck typing в Python
        - Абстрактные классы (ABC, abstractmethod)
        Практика: абстрактный класс Storage с методами save/load,
                  реализации: FileStorage, MemoryStorage

[ ] 1.5  Dataclass и __slots__
        - dataclass как упрощённый способ создания классов данных
        - __slots__ для оптимизации памяти
        - Сравнение: обычный класс vs dataclass vs Pydantic BaseModel
        Практика: переписать класс User тремя способами и сравнить


##### ЭТАП 2 — PYDANTIC И ВАЛИДАЦИЯ ДАННЫХ #####
Pydantic — сердце FastAPI. Без него невозможно работать с данными.

[ ] 2.1  Pydantic BaseModel — основы
        - Создание моделей, типизация полей
        - Валидация входных данных (автоматическая)
        - Значения по умолчанию, Optional, Field()
        Практика: модель UserCreate(name, email, age) с валидацией

[ ] 2.2  Вложенные модели и связи
        - Модель внутри модели (Address внутри User)
        - Списки моделей (List[Item])
        - model_dump(), model_json_schema()
        Практика: модель Order с вложенным списком Product

[ ] 2.3  Кастомная валидация
        - @field_validator — валидация отдельного поля
        - @model_validator — валидация всей модели
        - Практика: валидация email (regex), проверка что password и confirm совпадают

[ ] 2.4  Разделение моделей: Create / Read / Update
        - Зачем нужны разные схемы для одной сущности
        - UserCreate (без id) → UserRead (с id) → UserUpdate (Optional поля)
        - Наследование моделей для переиспользования полей
        Практика: полный набор схем для сущности Product


##### ЭТАП 3 — FASTAPI: МАРШРУТЫ И СТРУКТУРА #####
Переходим от одного файла к правильной архитектуре.

[ ] 3.1  HTTP-методы и эндпоинты
        - GET, POST, PUT, PATCH, DELETE — когда какой использовать
        - Path-параметры: /users/{user_id}
        - Query-параметры: /users?skip=0&limit=10
        - Body-параметры: приём JSON через Pydantic-модели
        Практика: CRUD-эндпоинты для сущности User (пока в одном файле)

[ ] 3.2  Response Model и статус-коды
        - response_model=UserRead — контроль того, что отдаём клиенту
        - status_code=201 для POST, 204 для DELETE
        - HTTPException — как правильно возвращать ошибки
        Практика: добавить response_model и обработку ошибок к CRUD

[ ] 3.3  APIRouter — разделение на модули
        - Один файл = один ресурс (users.py, products.py)
        - router = APIRouter(prefix="/users", tags=["Users"])
        - Подключение роутеров в main.py через app.include_router()
        Практика: разделить приложение на роутеры users и products

[ ] 3.4  Структура проекта (ВАЖНО!)
        Целевая структура:
        app/
        ├── main.py              # точка входа, подключение роутеров
        ├── config.py            # настройки (из .env)
        ├── models/              # ORM-модели (SQLAlchemy)
        │   ├── user.py
        │   └── product.py
        ├── schemas/             # Pydantic-схемы
        │   ├── user.py
        │   └── product.py
        ├── routers/             # Эндпоинты (APIRouter)
        │   ├── user.py
        │   └── product.py
        ├── services/            # Бизнес-логика (ООП!)
        │   ├── user_service.py
        │   └── product_service.py
        ├── repositories/        # Работа с БД (ООП!)
        │   ├── base.py
        │   └── user_repository.py
        └── database.py          # Подключение к БД
        Практика: перенести текущий код в эту структуру


##### ЭТАП 4 — БАЗА ДАННЫХ (SQLAlchemy + ООП) #####
Бекенд без базы данных — не бекенд.

[ ] 4.1  SQLAlchemy ORM — основы
        - Engine, Session, declarative_base
        - Создание ORM-моделей (class User(Base))
        - Типы полей: Integer, String, Boolean, DateTime
        - Создание таблиц (Base.metadata.create_all)
        Практика: ORM-модель User + подключение к SQLite

[ ] 4.2  CRUD через SQLAlchemy
        - session.add(), session.query(), session.commit()
        - Фильтрация, сортировка, пагинация
        - Связь ORM-модели с Pydantic-схемой
        Практика: полный CRUD для User через SQLAlchemy

[ ] 4.3  Alembic — миграции базы данных
        - Зачем нужны миграции (вместо create_all)
        - alembic init, alembic revision --autogenerate
        - alembic upgrade head, alembic downgrade
        Практика: добавить поле phone к User через миграцию

[ ] 4.4  Отношения между таблицами
        - One-to-Many (User → Posts)
        - Many-to-Many (User ↔ Roles)
        - relationship(), ForeignKey, back_populates
        Практика: User имеет много Posts, Post принадлежит User


##### ЭТАП 5 — ООП-ПАТТЕРНЫ В БЕКЕНДЕ (ключевой этап) #####
Здесь объединяются ООП и бекенд-архитектура.

[ ] 5.1  Repository Pattern (паттерн Репозиторий)
        - Абстрактный BaseRepository с методами: get, get_all, create, update, delete
        - Конкретные реализации: UserRepository, ProductRepository
        - Репозиторий = единственное место, где живут SQL-запросы
        Практика: абстрактный репозиторий + UserRepository

[ ] 5.2  Service Layer (слой сервисов / бизнес-логики)
        - UserService принимает UserRepository через конструктор
        - Сервис содержит бизнес-правила (не SQL!)
        - Пример: "нельзя создать пользователя с занятым email"
        Практика: UserService с валидацией бизнес-правил

[ ] 5.3  Dependency Injection в FastAPI
        - Depends() — встроенный DI-контейнер FastAPI
        - Инъекция сессии БД → репозитория → сервиса → роутера
        - Зачем это нужно: тестируемость, слабая связанность
        Практика: связать Repository → Service → Router через Depends

[ ] 5.4  Unit of Work (опционально, продвинутый уровень)
        - Управление транзакциями через один объект
        - Когда пригодится: операции с несколькими таблицами
        Практика: UnitOfWork для операции "создать заказ + списать товар"

[ ] 6.3  Защита эндпоинтов

        - Ролевая модель: admin, user, moderator
